## Table Stakes

### 1. Database Migration: MySQL to Supabase PostgreSQL

| Feature | Why Expected | Complexity | Dependencies on Existing Code |
|---------|--------------|------------|-------------------------------|
| Replace `aiomysql` driver with `asyncpg` in `database.py` | Current driver is MySQL-only; PostgreSQL requires asyncpg | Low | `database.py` URL parsing (lines 7-9), `requirements.txt` |
| Fix `server_default="0"` on Boolean columns | MySQL accepts `"0"` for false; PostgreSQL Boolean requires `text("false")` | Low | `models.py` line 82: `skipCache` uses `server_default="0"` |
| Handle `onupdate=func.now()` removal | SQLAlchemy `onupdate` is ORM-level, not server-side. Works with MySQL but PostgreSQL has no native `ON UPDATE CURRENT_TIMESTAMP`. Must add DB trigger or move to application-level `updated_at` setting | Med | 4 models use `onupdate=func.now()`: User, Video, Analysis, Statistic |
| Fix Enum handling for PostgreSQL | PostgreSQL creates real ENUM types (not VARCHAR). Alembic does not auto-manage PG enum creation -- requires explicit `CREATE TYPE` in migrations | Med | 3 enums: `UserRole`, `PipelineMode`, `ProcessingStatus` in models.py |
| Validate JSON columns (consider JSONB) | PostgreSQL JSONB is preferred for indexing and operators. Current `JSON` type works but JSONB is the standard choice for PG | Low | 8 JSON columns across Event, Track, Statistic, Commentary, Analysis |
| Update connection string parsing in `database.py` | Must handle `postgresql://` and `postgresql+asyncpg://` URL schemes (replace current `mysql://`/`mysql2://` logic) | Low | `database.py` lines 7-9 |
| Data migration strategy | Either migrate existing dev data or use clean-slate approach with Alembic creating fresh schema on Supabase | Low | All 7 tables in models.py |
| Leverage `RETURNING` clause benefits | PostgreSQL supports `RETURNING` on INSERT -- SQLAlchemy 2.0+ uses this automatically with asyncpg, making post-commit `db.refresh()` calls unnecessary | Low | Every router that does `db.commit()` then `db.refresh()` |

**What "done" looks like:** App starts with `postgresql+asyncpg://` URL pointing at Supabase. All CRUD operations pass. All existing E2E tests pass unchanged.

## Anti-Features

Features to explicitly NOT build during this milestone.

| Anti-Feature | Why Avoid | What to Do Instead |
|--------------|-----------|-------------------|
| OAuth/SSO integration | Out of scope per PROJECT.md; auto-login dev mode stays | Keep AutoLoginMiddleware for dev; add conditional JWT decode path for production |
| Supabase Auth integration | Couples to Supabase-specific auth SDK; this milestone is database migration only | Use standard JWT validation; Supabase Auth is a future milestone |
| Supabase Storage for videos | Separate concern, large change surface | Keep local file storage; migrate storage separately |
| Supabase Realtime (replace WebSocket) | Working FastAPI WebSocket implementation exists and is tested | Keep FastAPI native WebSocket |
| New user-facing features | Explicitly out of scope -- structural hardening only | Defer to v0.3 |
| CV pipeline algorithm changes | Pipeline must remain functionally identical per constraints | Only change model URL sourcing (env vars), not algorithm code |
| ORM migration to SQLModel | Unnecessary rewrite; SQLAlchemy 2.0 mapped_column is already modern | Keep SQLAlchemy, it is more mature and better documented |
| Column rename (camelCase to snake_case) | Would break all frontend API calls and require schema migration | Keep camelCase column names; address in a future API versioning milestone |
| Production data migration tooling | No production deployment exists -- this is a dev-stage project | Use Alembic `upgrade head` for fresh schema; skip data migration |
| Full E2E test rewrite | 4 Playwright specs already exist and work | Add unit tests alongside existing E2E; keep Playwright as-is |

## Feature Dependencies

```
CDN Removal ──────────────────────────────> (independent, do first)

Database Migration (asyncpg) ─────┐
                                  ├──> Alembic Setup (needs PG-compatible models)
Model PG Compatibility Fixes ─────┘

Alembic Setup ────────────────────────────> Backend Tests (tests need migration-managed DB)

Analysis.tsx Decomposition ───────────────> Frontend Tests (test decomposed components)

Security Hardening (JWT path) ────────────> Backend Tests (test both auth paths)

Linting Setup ────────────────────────────> (independent, do anytime)
```

Key ordering constraints:
1. **CDN removal** is independent and low-risk -- do first to remove external dependencies early
2. **Database migration** must precede **Alembic setup** because Alembic config depends on PostgreSQL-compatible models
3. **Component decomposition** must precede **frontend unit tests** because testing a 2358-line monolith is impractical
4. **Security hardening** should precede **backend tests** so tests can validate both dev and production auth paths
5. **Linting** is independent and can run in parallel with anything

---

## MVP Recommendation

Prioritize in this order:

1. **CDN Removal** -- Eliminates external dependency risk immediately; lowest complexity; highest independence
2. **Database Migration + Alembic** -- Core infrastructure change everything else depends on; do together as one phase
3. **Security Hardening** -- Must happen before any deployment; informs test structure for auth paths
4. **Analysis.tsx Decomposition** -- Enables frontend testing; reduces maintenance burden for all future work
5. **Testing Infrastructure** -- Builds on stable foundation of migrated DB + decomposed components + security paths
6. **Linting** -- Can be interleaved anywhere but formally last since it is purely additive quality

Defer to future milestones: **Supabase RLS**, **Supabase Auth**, **Supabase Storage**, **API versioning**, **column rename**

## Codebase-Specific Gotchas Identified

These are concrete issues found during code inspection that will surface during implementation:

| Issue | Location | Impact | Resolution |
|-------|----------|--------|------------|
| `server_default="0"` on Boolean | `models.py:82` | PostgreSQL rejects `"0"` as Boolean default | Change to `server_default=text("false")` |
| `onupdate=func.now()` on 4 models | `models.py:45,66,91,160` | ORM-level only; no PG equivalent without trigger | Add PG trigger in Alembic migration OR set `updated_at` in application code before commit |
| MySQL URL prefix detection | `database.py:7-9` | Hardcoded `mysql://` and `mysql2://` replacement | Replace with `postgresql://` to `postgresql+asyncpg://` logic |
| `event_metadata` column alias | `models.py:111` | Uses `mapped_column("metadata", ...)` because `metadata` is reserved in SQLAlchemy -- this works fine in PG but verify Alembic autogenerate picks up the column name correctly | Test during initial migration generation |
| No FK constraints in models | All models | `userId`, `videoId`, `analysisId` are plain Integer columns with no ForeignKey -- works but Alembic autogenerate won't create FK constraints | Decide: add ForeignKey declarations now (recommended) or defer |
| `async_session` can be None | `database.py:15` | If DATABASE_URL is empty, session is None -- guards exist in `deps.py` but not all code paths check | Ensure Supabase URL is always set; consider removing None path |
